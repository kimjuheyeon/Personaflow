# AI 기반 자동 유저테스트 플랫폼
## 서비스 기획안 v1.2
> SaaS 보편 평가 구조 기반 MVP — 웹 플랫폼 → Figma 플러그인 확장

---

## 1. 서비스 개요

### 1.1 배경 및 목적

기존 유저테스트(UT)는 실제 유저 섭외, 인터뷰 일정 조율, 결과 분석까지 최소 수 일에서 수 주가 소요되며 비용도 상당하다. 이로 인해 초기 시안 단계에서는 UT를 건너뛰거나, 디자이너의 주관적 판단에만 의존하는 경우가 많다.

본 플랫폼은 AI 페르소나가 실제 유저처럼 제품 시안을 탐색하고, 수 분 내에 구조화된 UX 리포트를 자동 생성함으로써 이 과정을 대폭 단축한다.

### 1.2 핵심 가치

- 시안 단계에서 즉시 피드백 — 이미지만 있어도 분석 가능
- 페르소나 기반 다각도 평가 — 직접 설정 또는 AI 자동 추천
- SaaS 보편 6축 평가 구조 — 어떤 제품에도 적용 가능한 표준화된 기준
- 명확한 심각도 분류 — Critical / Warning / Suggestion 3단계
- 대화형 UT — 화면별 인라인 채팅으로 핀포인트 질문 가능

## 2. 서비스 플로우

총 4단계로 구성되며, 각 단계는 순차적으로 진행된다.

| 단계 | 항목 | 설명 |
|---|---|---|
| ① 시안 입력 | 입력 방식 | 이미지 업로드(PNG/JPG) 또는 프로토타입 URL |
| | 멀티 화면 | 여러 화면을 flow로 연결해 입력 가능 |
| ② 페르소나 설정 | 직접 설정 | 직무, 기술 수준, 목표, 사용 맥락 등 직접 입력 |
| | AI 추천 | 제품 분석 후 적합한 페르소나 3~5개 자동 생성 및 선택 |
| ③ AI 테스트 실행 | 6축 평가 | SaaS 보편 평가 기준으로 페르소나 관점의 UX 시뮬레이션 |
| | 감정 반응 | 긍정 / 혼란 / 이탈 순간 기록 |
| ④ 리포트 출력 | 심각도 분류 | Critical / Warning / Suggestion 3단계로 문제 분류 |
| | Export | PDF, Figma 코멘트, Notion, JSON 형태로 내보내기 |

## 3. AI 테스트 엔진 — SaaS 보편 6축 평가

### 3.1 평가 방법론 기반

두 가지 방법론을 결합해 평가 근거를 구조화한다.

- 닐슨의 10가지 휴리스틱 — 각 축의 세부 체크리스트 근거로 활용
- 태스크 기반 평가 — 완료율, 클릭 수, 오류율 등 정량 지표로 심각도 판단

AI는 각 축을 평가할 때 어떤 휴리스틱 원칙을 위반했는지와 태스크 지표로 얼마나 심각한지를 함께 출력한다.

**리포트 출력 예시**

```
[Warning] 핵심 태스크 완료 — ③축
위반 원칙: 닐슨 #1 시스템 상태 가시성
문제: 파일 업로드 후 처리 중 상태를 알 수 없음
태스크 지표: 평균 대기 후 이탈 발생 (클릭 수 +3회 초과)
개선 제안: 업로드 진행률 표시 또는 완료 피드백 추가
```

### 3.2 6축 × 방법론 매핑

| 6축 | 닐슨 휴리스틱 (근거) | 태스크 기반 측정 지표 |
|---|---|---|
| ① 첫인상 & 가치 전달 | #2 실제 세계와의 일치, #8 미적이고 최소한의 디자인 | 5초 내 목적 파악 여부 |
| ② 온보딩 흐름 | #6 기억보다 인식, #5 오류 방지 | 첫 태스크 완료율, 이탈 발생 지점 |
| ③ 핵심 태스크 완료 | #1 시스템 상태 가시성, #9 오류 인식·복구 | 완료율, 소요 클릭 수, 오류율 |
| ④ 내비게이션 구조 | #3 유저 통제 및 자유, #4 일관성과 표준 | 이탈 후 복귀 성공률, 길 잃음 횟수 |
| ⑤ 정보 구조 & 용어 | #2 실제 세계와의 일치, #6 기억보다 인식 | 용어 혼란으로 인한 정지 횟수 |
| ⑥ 전환 & 재방문 유도 | #7 유연성과 효율성, #10 도움말과 문서 | CTA 클릭률, 전환 경로 완료율 |

### 3.3 각 축 상세

**① 첫인상 & 가치 전달**

제품을 처음 접했을 때 5초 내에 무엇을 하는 서비스인지 파악할 수 있는가를 평가한다.

- 5초 내 제품 목적 파악 여부
- 헤드라인 / 서브카피 명확성
- 핵심 CTA 시인성 및 위치 적절성
- 브랜드 신뢰감 (색상, 폰트, 레이아웃 인상)

**② 온보딩 흐름**

신규 유저가 처음 제품에 진입할 때 겪는 마찰을 평가한다.

- 회원가입 / 로그인 경로의 명확성
- 첫 사용 진입 난이도 (설정 없이 바로 시작 가능한가)
- 빈 상태(Empty state) UX의 안내 충분성
- 초기 설정 단계 수 및 복잡도

**③ 핵심 태스크 완료**

제품의 핵심 기능을 유저가 스스로 완료할 수 있는가를 평가한다.

- 주요 기능까지의 도달 경로 명확성
- 클릭 수 및 화면 뎁스 적절성
- 로딩 / 처리 중 피드백 인지 여부
- 오류 발생 시 복구 경로 안내

**④ 내비게이션 구조**

유저가 현재 위치를 인지하고 원하는 곳으로 이동할 수 있는가를 평가한다.

- 메뉴 레이블의 이해도
- 현재 위치 인지 (Breadcrumb, 활성 탭 등)
- 뒤로가기 / 이전 단계 복귀 UX
- 검색 기능 접근성

**⑤ 정보 구조 & 용어**

콘텐츠의 위계와 언어가 유저 수준에 맞게 설계되어 있는가를 평가한다.

- 레이블 및 용어의 명확성
- 콘텐츠 위계의 적절성 (중요도 순서)
- 도움말 / 툴팁의 필요성 및 위치
- 전문용어 과다 사용 여부

**⑥ 전환 & 재방문 유도**

유저가 계속 사용하고 싶어지도록 설계되어 있는가를 평가한다.

- 업그레이드 / 결제 전환 경로
- 알림, 이메일 등 훅 포인트 설계
- 성공 경험 (Aha moment) 설계
- 이탈 방지 장치 (저장, 진행 상태 유지 등)

## 4. 페르소나 시스템

### 4.1 직접 설정 모드

| 항목 | 예시 |
|---|---|
| 직무 / 역할 | UX 디자이너, 마케터, 개발자, 비전공 관리자 등 |
| 디지털 친숙도 | 초급 / 중급 / 고급 |
| 사용 목적 | 업무 자동화, 데이터 분석, 협업 도구 등 |
| 기기 환경 | 데스크탑 / 모바일 / 태블릿 |
| 사용 맥락 | 처음 써보는 유저 / 기존 도구 대체 검토 중 |

### 4.2 AI 추천 모드

시안을 분석해 해당 제품에 적합한 페르소나를 3~5개 자동 생성한다. 각 페르소나별 테스트 결과를 비교해 타겟 유저군에 따른 UX 차이를 파악할 수 있다.

## 5. 대화형 UT — 화면별 인라인 채팅

### 5.1 개념

AI가 일방적으로 분석하는 것에서 나아가, 디자이너가 특정 화면에 대해 직접 질문을 던지면 AI 페르소나가 해당 관점으로 답하는 대화형 구조. 분석 중 궁금한 지점을 핀포인트로 파고들 수 있다.

### 5.2 UI 구조

각 프레임 이미지 하단에 채팅 입력창이 인라인으로 붙는 방식. 리포트 전체가 아닌 해당 화면에 한정된 대화가 이루어진다.

```
┌─────────────────────────────┐
│        프레임 이미지          │
├─────────────────────────────┤
│ AI 분석 결과 (축약)           │
│ ⚠ Warning — CTA 시인성 낮음  │
├─────────────────────────────┤
│ 💬 이 화면에 대해 질문하기    │
│ ┌─────────────────────────┐ │
│ │ 이 버튼 레이블 직관적이야? │ │
│ └─────────────────────────┘ │
│                      [전송]  │
│                              │
│ 🤖 [페르소나명]              │
│ "비전공자 입장에서 '시작하기' │
│  보다 '무료로 써보기'가 더    │
│  와닿을 것 같아요."          │
└─────────────────────────────┘
```

### 5.3 질문 유형 예시

| 유형 | 예시 |
|---|---|
| 행동 예측 | "이 화면에서 유저가 제일 먼저 클릭할 것 같은 게 뭐야?" |
| 혼란 지점 확인 | "이 버튼 레이블이 직관적으로 느껴져?" |
| 페르소나 반응 | "비전공자 관리자라면 이 용어를 이해할 수 있을까?" |
| 대안 제시 요청 | "이 온보딩 흐름 말고 더 나은 방식이 있을까?" |
| 우선순위 확인 | "지금 발견된 문제 중에 제일 먼저 고쳐야 할 게 뭐야?" |

### 5.4 페르소나 유지 원칙

채팅 중에도 AI는 설정된 페르소나를 유지한다. "비전공자 관리자"로 테스트 중이면 채팅 답변도 그 관점에서만 나와야 한다. 페르소나를 벗어난 일반적인 UX 조언은 제공하지 않는다.

### 5.5 로드맵

- MVP — 프레임별 인라인 채팅, 페르소나 유지 대화
- 2단계 — 채팅 내용을 리포트에 자동 병합, 대화 히스토리 Export

## 6. 리포트 출력 구조

### 6.1 심각도 분류

| 등급 | 기준 | 대응 |
|---|---|---|
| 🔴 Critical | 태스크 완료 자체가 불가능한 문제 | 즉각 수정 필요 |
| 🟡 Warning | 완료는 되지만 마찰이 존재하는 문제 | 개선 권고 |
| 🔵 Suggestion | 경험을 향상시킬 수 있는 기회 | 선택적 개선 |

### 6.2 리포트 구성 항목

- 평가 요약 — 6축 점수 및 전체 총평
- 문제 목록 — 심각도별 정렬, 위반 휴리스틱 원칙 명시
- 태스크 지표 — 완료율, 클릭 수, 오류율, 이탈 지점
- 페르소나별 비교 — 복수 페르소나 선택 시 차이 비교
- 인라인 채팅 히스토리 — 화면별 대화 내용 포함 (2단계)
- 개선 제안 — 각 문제에 대한 구체적 UX 개선안
- Export — PDF / Figma 코멘트 / Notion / JSON

## 7. 인터랙션 흐름 전달 방식

AI가 단순 화면 분석을 넘어 실제 유저의 탐색 흐름을 시뮬레이션하려면, 화면 간 인터랙션 연결 정보가 필요하다.

### MVP (1단계) — 유저 직접 흐름 정의

유저가 프레임 이미지를 업로드한 뒤, 화면 간 연결 관계를 직접 매핑하는 방식이다.

**동작 방식**

- 여러 프레임 이미지를 순서대로 업로드
- 각 화면에서 어떤 요소를 클릭하면 어디로 이동하는지 유저가 직접 연결
- 연결 정보를 AI에 전달해 흐름 기반 UX 시뮬레이션 실행

**흐름 정의 예시**

| 출발 화면 | 트리거 | 도착 화면 |
|---|---|---|
| 화면 A — 랜딩 | "시작하기" 버튼 클릭 | 화면 B — 회원가입 |
| 화면 B — 회원가입 | 폼 제출 (성공) | 화면 C — 대시보드 |
| 화면 B — 회원가입 | 폼 제출 (실패) | 화면 D — 에러 상태 |
| 화면 C — 대시보드 | "새 프로젝트" 클릭 | 화면 E — 프로젝트 생성 |

**장단점**

- 장점 — 구현 난이도 낮음, 연결 정확도 높음, 즉시 적용 가능
- 단점 — 유저 추가 작업 필요, 화면 수가 많을수록 매핑 부담 증가

### 디벨롭 (2단계) — Figma API 자동 파싱

Figma API의 `reactions` 데이터를 활용해 프로토타입 링크 하나만 입력하면 화면 간 인터랙션 연결 관계를 자동으로 추출한다.

**동작 방식**

- Figma 프로토타입 링크 입력
- API를 통해 프레임 이미지 + reactions(연결 관계) 자동 추출
- 파싱된 흐름 정보를 AI에 전달해 자동 시뮬레이션 실행
- 파싱 불가한 복잡한 조건부 흐름은 유저 보완 입력으로 처리

**reactions 구조 예시**

```json
{
  "reactions": [
    {
      "trigger": { "type": "ON_CLICK" },
      "action": { "type": "NODE", "destinationId": "프레임ID" }
    }
  ]
}
```

**장단점**

- 장점 — 유저 추가 작업 없음, Figma 프로토타입 그대로 활용
- 단점 — Figma API 연동 개발 필요, 복잡한 조건부 인터랙션 파싱 한계

## 8. 플랫폼 확장 전략

### 8.1 단계별 플랫폼 전략

| 단계 | 플랫폼 | 특징 |
|---|---|---|
| MVP | 웹 플랫폼 | 브라우저 기반, 누구나 URL로 접근 가능, 빠른 검증 |
| 확장 | Figma 플러그인 | 디자이너 작업 흐름 안에서 바로 실행, Figma 데이터 직접 접근 |

### 8.2 기술 구조 — 백엔드 분리 원칙

웹과 Figma 플러그인은 UI 환경이 완전히 다르지만, AI 분석 핵심 로직은 동일한 백엔드 API 서버를 공유한다.

```
웹 MVP (브라우저)          Figma 플러그인 (iframe)
        ↓                          ↓
     ┌─────────────────────────────────┐
     │       AI 분석 백엔드 API 서버     │
     │  - 6축 UX 평가 엔진             │
     │  - 페르소나 시뮬레이션           │
     │  - 리포트 생성기                │
     └─────────────────────────────────┘
```

| 구분 | 웹 MVP | Figma 플러그인 |
|---|---|---|
| UI 레이어 | 브라우저 (React 등) | 플러그인 UI (iframe 기반) |
| Figma 데이터 접근 | API 호출 또는 수동 업로드 | 플러그인 내부 직접 접근 |
| AI 분석 엔진 | 백엔드 API 서버 공유 | 백엔드 API 서버 공유 |
| 인터랙션 흐름 | 유저 직접 매핑 (MVP) | reactions 자동 파싱 |
| 리포트 출력 | 웹 페이지 내 렌더링 | Figma 코멘트 직접 삽입 |

### 8.3 Figma 플러그인 전환 시 핵심 이점

- 디자이너가 Figma를 벗어나지 않고 UT 실행 가능
- 프로토타입 reactions 자동 파싱으로 흐름 정의 작업 제거
- 리포트 결과를 Figma 코멘트로 직접 삽입해 팀 공유 즉시 가능
- 선택한 프레임만 부분 테스트 가능

## 11. 기술 스택 제안 (CLI 기반 구축)

### 프론트엔드

```
React (Vite)
- 빠른 번들링, 플러그인 전환 시 UI 재사용 용이
- Figma 플러그인도 iframe 기반 React 그대로 사용 가능
```

### 백엔드

```
Node.js + Express
- AI 분석 엔진 API 서버
- 웹 / 플러그인 양쪽에서 동일하게 호출
```

### AI

```
Anthropic Claude API
- Vision: 이미지 기반 화면 분석
- claude-sonnet-4-6 모델 사용
- 6축 평가 프롬프트 + 페르소나 주입
```

### CLI 초기 세팅 순서

```bash
# 1. 프로젝트 초기화
mkdir ai-ut-platform && cd ai-ut-platform
npm init -y

# 2. 백엔드 세팅
mkdir -p packages/backend && cd packages/backend
npm init -y
npm install express @anthropic-ai/sdk dotenv multer cors

# 3. 웹 프론트엔드 세팅
cd ../..
npm create vite@latest packages/web -- --template react
cd packages/web && npm install

# 4. 환경변수 설정
echo "ANTHROPIC_API_KEY=your_key_here" > packages/backend/.env

# 5. 백엔드 실행
cd packages/backend && node index.js

# 6. 웹 실행
cd packages/web && npm run dev
```
